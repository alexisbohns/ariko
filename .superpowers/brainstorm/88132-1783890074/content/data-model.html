Data model — the vault

Your existing molecule → atom → version stays intact. We add a raw Capture in front of it, plus shared building blocks for media, bilingual text, and provenance.

Shared building blocks (used by everything below)
LocalizedText
{ en?, fr? }
For names & descriptions. Lets pbbls store FR + EN release notes side by side; plain string is fine for a song.
Media[]
embed → { provider, url } (SoundCloud, Spotify, Deezer, Ausha, YouTube, Vimeo, Figma)
image → { storageKey, url, alt, w, h } (your uploads, in object storage — never the repo)
Source (provenance)
{ kind, url?, externalId?, capturedAt }
kind = manual · github · changelog · arkaik · … So you always know where a thing came from + can dedupe.

One real fork for you

Should a raw Capture be its own thing, or just an early-stage Version?

A

Capture is its own entity ★ recommended

Raw inbox junk stays out of the clean atomic model until you triage it. One capture can become several versions (a PR touching two features), or be discarded. Your molecule/atom/version data never gets polluted with half-baked drafts.

B

Captures are just draft Versions

Simpler — one fewer entity. An inbox item is a Version with status: inbox and no parent yet. But drafts and real work share a table, and the 1-capture-→-many-versions case gets awkward.
